# "My puzzles" creator-feedback view, per-puzzle solve counts, and a discovery lane for the community stream

This change closes the creator loop for UGC: every submission now tracks a `solves` count alongside `votes`, a new "My puzzles" scene shows a builder their puzzles with headline totals, and the community stream gains a third "discovery" lane that randomly resurfaces older mid-tier puzzles so they stop getting buried behind the top-voted and newest windows. The solve count is wired through `recordPlayed` (first-solve-only), surfaced as social proof on the community win screen, and aggregated by a pure `creatorTotals`. The stream builder now attaches live votes+solves to every candidate and the round-robin interleaver was generalized from two lanes to three. Submissions are additionally indexed per creator (`ugc:bycreator:{user}`) to power "My puzzles", and the editor lets a guest build/test freely, stashing their board to `localStorage` before a sign-in reload so publishing survives the round-trip.

Watch for: the per-puzzle solve counter is incremented under a non-atomic check-then-act (concurrent solves can double-count, *likely*), and a creator solving their own puzzle in the stream inflates its solve count even though self-*votes* are explicitly blocked (*confirmed*). Separately, building a stream now issues up to ~180 individual `zScore` round-trips for live vote counts because there is no batch `zmScore` (*confirmed*, perf), and `/api/ugc/list` is now re-hit on every end-of-queue refetch.

## High-level view

The solve counter is the headline creator-feedback signal, and it's written with a read-then-conditionally-write sequence (`zScore` then `hIncrBy`) that isn't atomic; two concurrent solves of the same puzzle by the same user can both miss the guard and double-count. The same `recordPlayed` path runs for a creator solving their own puzzle, so self-solves count — inconsistent with the explicit no-self-vote rule next door.

Live vote counts are read one id at a time. Solves get a single `hMGet`, but votes have no batch equivalent, so every candidate id outside the top-pool seed gets its own `zScore` in a `Promise.all`. On a healthy catalog that's the recent window (up to 150) plus the discovery sample (30) per stream build, and the stream is now rebuilt on every end-of-queue refetch as well as on initial load.

The three-lane round-robin terminates correctly and the new required `solves` field is safe for legacy records (it's always overwritten from the hash, never read from stored JSON), so old submissions can't leak `undefined`. The discovery lane is sound but in production it's always a subset of the resolved pool, which the unit test doesn't model — it tests an additive case that can't occur live.

The social-proof numbers on the win screen are a load-time snapshot carried through the registry: they never include the solve that just happened and can be stale. The editor's guest-publish flow preserves work across the sign-in reload via `localStorage`, but the stash has no expiry and is restored on the *next* editor open regardless of whether sign-in actually happened, so a later fresh build can be overwritten by a stale draft.

### Solve counter: the first-solve guard isn't atomic, and self-solves count

`recordPlayed` gained a per-puzzle solve tally meant to tick once per distinct solver:

```ts
const alreadySolved = await redis.zScore(keys.ugcPlayed(username), id);
await redis.zAdd(keys.ugcPlayed(username), { member: id, score: Date.now() });
if (typeof alreadySolved !== 'number') {
  await redis.hIncrBy(keys.ugcSolves(), id, 1);
}
```

The played-set `zAdd` is idempotent, but the counter is guarded by a prior read, not by the write itself. Two `recordPlayed` calls for the same user+puzzle that interleave before either `zAdd` lands both observe `alreadySolved === undefined` and both `hIncrBy`. `markPlayed` is fired from the client as an un-awaited `void this.markPlayed(...)` on the community win, so retries or a double-trigger aren't serialized client-side either. The trigger is narrow (same user winning the same board near-simultaneously, e.g. two tabs) but the counter is the headline creator-feedback number, so the failure mode is upward drift. An atomic guard removes the window: `hSetNX` a per-user marker keyed to the puzzle and `hIncrBy` only when it was newly set (*likely*).

A second, separate issue in the same path: `onWinCommunity` calls `markPlayed` unconditionally, including when `isMine` is true. Own puzzles are intentionally kept in the creator's own stream, so a creator can solve their own puzzle there, and that solve increments `ugcSolves`. This is inconsistent with votes, where `votePuzzle` explicitly rejects `sub.creator === username`. The "My puzzles" totals and the win-screen social-proof line will therefore include the creator's own solves. If self-solves shouldn't count, skip the `hIncrBy` (keep the played-set add — you still want it to leave their stream) when `creator === username` (*confirmed*).

### Vote lookups cost N round-trips; solves cost one

Solves and votes are fetched with different strategies because of what the Devvit redis client offers. Solves use a single batched `hMGet` over the `ugc:solves` hash. Votes have no batch `zmScore`, so `votesByID` seeds from the top-pool scores it already has and then issues one `zScore` per remaining id:

```ts
const missing = ids.filter((id) => !map.has(id));
const scores = await Promise.all(missing.map((id) => redis.zScore(keys.ugcIndex(), id)));
```

The seed covers the top pool (up to `POOL_SIZE` = 150). Everything in the recent window that isn't also top-voted, plus the 30-id discovery sample, falls into `missing` — on a catalog larger than the windows, on the order of 150 + 30 individual Redis calls per stream build, fanned out in parallel. It's the stream's dominant cost and scales with catalog size, and it now runs more often: `nextCommunity` re-hits `/api/ugc/list` on every end-of-queue refetch, not just on the initial `CommunityScene` load. If vote freshness on the non-top lanes can tolerate slight staleness, storing `votes` in the submission JSON (refreshed on vote) and doing live lookups only for the top lane would collapse this to roughly one `mGet` (*confirmed*, perf).

### The new `solves` field is safe for legacy records

Both `listStreamForUser` and `listMine` overwrite `solves` (and `votes`) from the live hash/sorted-set rather than trusting the parsed JSON —

```ts
const sub: UgcSubmission = { ...parsed, votes: voteByID.get(ids[i]) ?? 0, solves: solveByID.get(ids[i]) ?? 0 };
```

so a submission stored before this change (no `solves` key) surfaces as `0`, never `undefined`. The only other readers of stored submissions are `submitPuzzle` (constructs fresh, includes `solves: 0`) and `votePuzzle` (reads `creator` only, doesn't echo the object back). No path returns a stored `UgcSubmission` to a client without the override, so the required-field promise holds at runtime — directly answering the "where could `solves` be missing" question: nowhere reachable.

### Three-lane round-robin ordering

The interleaver went from a two-cursor boolean toggle to an n-lane round-robin with a no-progress counter:

```
lanes = [byVotes, byNew, discovery]
rotate lane each step; advance that lane's cursor past anything already `seen`
place one if available (reset sinceProgress) else sinceProgress++
stop when out.length == limit OR sinceProgress == lanes.length
```

Termination is sound: cursors are monotonic, `sinceProgress` only resets on a placement, and it stops once a full cycle of lanes yields nothing — no infinite loop, and no premature stop while any lane still has an unseen item. `sampleIdRange` is inclusive of `1..maxId` and `maxId` matches the counter's last-issued id (`incrBy` returns the new value, used directly as the id), so no off-by-one in what's sampled, and gaps from missing ids are tolerated downstream (`mGet` null → skipped, discovery `filter(defined)`).

One nuance the tests don't capture: in production the discovery candidates are resolved from the same id set as `subs` (`discovery = sampleIds.map(byId.get).filter(defined)`), so every discovery item is already in `byVotes`/`byNew`. The lane therefore isn't additive — it's a reordering that gives sampled ids a third, earlier slot in the rotation so they surface within `limit` instead of sitting deep in the vote/recency tail. The unit test feeds a discovery id (`'9'`) that is *not* in `subs`, exercising an additive path that can't happen live. A test where discovery ⊆ subs would lock in the actual intended behavior (a low-vote, old item appearing earlier than its vote/recency rank alone would place it).

### Social-proof numbers are a load-time snapshot

The win-screen votes/solves come from `this.communityPuzzle`, populated when the stream batch is built and carried untouched through the registry queue into each `GameScene`. So the number shown is whatever was true when the batch loaded: it never includes the solve the player just completed (a freshly-created 0-solve puzzle shows no "solves" line to its first solver; a 3-solve puzzle still reads "3 solves" after this player makes it 4), and it can be stale if others voted/solved since the batch loaded. The code comment ("from the stream-load snapshot") shows this is deliberate and it's only cosmetic, but it's player-visible on a screen meant to convey momentum. An optimistic `+1` to `solves` on the local solve would read more truthfully without a refetch (*confirmed*, cosmetic).

### End-of-queue refetch and the fire-and-forget played race

When the queue is exhausted, `nextCommunity` refetches a fresh batch and filters only the just-finished id:

```ts
const more = await this.fetchCommunityBatch(this.communityPuzzle?.id);
// fetchCommunityBatch: data.submissions.filter((s) => s.id !== excludeId)
```

The explicit filter covers the immediate back-to-back repeat; everything else solved earlier this session is expected to be gone via the server's `playedIds`. But `markPlayed` is fire-and-forget, so if an earlier solve's POST is still in flight or failed, that puzzle's `ugc:played` entry isn't there yet and it can reappear in the new batch. Earlier POSTs almost always land before you reach the end of a 20-item queue, so likelihood is low; the honest gap is that a *failed* `markPlayed` silently re-shows a solved puzzle rather than being retried. Since skips don't mark-played anyway, re-surfacing a solved puzzle is a soft failure, not a dead end (*likely*, low).

### Guest publish: draft stashing and the reload round-trip

`showLoginPrompt()` reloads the page and wipes the Phaser registry, so the editor stashes the in-progress board to `localStorage` (try/catch-wrapped, which matters inside Reddit's iframe) before prompting, and `init` restores it via `this.registry.get('editor.draft') ?? loadPendingDraft()`. Two rough edges:

The stash has no expiry and is restored on the *next* editor open unconditionally — it isn't scoped to "we just came back from a sign-in reload." If the guest cancels the login, the draft lingers; the next time they open Build (a deliberate fresh start, possibly days later) the old board silently loads and `touched` is set true. It's cleared on restore so it only resurfaces once, but a user intending a clean build gets a stale one. Timestamping the stash and ignoring it past a short window, or restoring only behind an explicit "returning from sign-in" flag, would tighten this (*confirmed*, low UX).

Also, after the sign-in reload nothing routes the user back to the editor (Boot opens the daily `GameScene`), and the restored draft still needs a second manual Submit. Work is preserved, but the flow isn't the auto-resume the stash might imply — worth setting expectations or routing back to Build post-reload (*confirmed*, UX).

### Resize, call-site, and destroyed-object checks (no issue found)

The explicit lifecycle asks check out. The win path now hides `hudText`/`hintText` (they sit in `uiLayer`, above the win card in `fxLayer`); a resize on the win screen doesn't undo that, because `updateHud` never re-shows `hudText` and re-gates `hintText` on `!this.won`. Both `community: {...}` construction sites and both now-async `nextCommunity` callers (`void this.nextCommunity()`) were updated together. `MyPuzzlesScene.load()` guards its post-fetch rebuild with `this.sys.isActive()`, so a player who leaves before the fetch resolves doesn't rebuild a torn-down scene, and its resize listener is removed on `shutdown`. All three TS projects type-check clean and the community suite passes.

### Test coverage

`creatorTotals` is covered for both the populated sum and the empty-creator zero case, and the discovery lane has tests for weave position, cross-lane de-duplication, and played/own-creator filtering. The 10-test community suite passes.

Not tested: the discovery test models discovery items absent from `subs` (impossible in production, see above) rather than the realistic subset case; the server `recordPlayed` first-solve-only and self-solve behavior has no test (server logic generally isn't unit-tested in this repo, so this is consistent, just unguarded); and `listMine`'s solve/vote override + sort isn't exercised.
