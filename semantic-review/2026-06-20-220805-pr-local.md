# Endless fast-fallback generation, load-failure recovery, and sign-in CTAs

This local diff (working tree vs `HEAD`, 5 code files + 3 docs) hardens the Endless
mode against the freeze that an on-the-fly graded generation could cause, and adds
client polish around failure and sign-in. The server now has two endless generators:
`generateEndlessPuzzle` (quality, off the request path) and `generateEndlessFallback`
(fast/bounded, on the request path); `/api/endless` switched its pool-empty branch to
the fallback. The solver budget (`maxStates`) is now a threaded `GenOption` (default
`60_000`) so endless can run the solver cheaper without touching the daily. On the
client, a load-failure recovery UI (`showLoadError`/`clearLoadError` + a
`fetchWithTimeout` abort), a logged-out "sign in" CTA on win screens, and a native
share sheet (with clipboard fallback) were added.

Watch for: the Endless win screen stacks five elements plus a sign-in button between
`h*0.44` and `h*0.8`; in the logged-out hint branch the lifetime-count text (`h*0.56`)
and the sign-in button (`h*0.61`) are only `0.05h` apart, which can crowd or overlap
44-48px touch targets on short mobile viewports (likely). Everything else is nits: a
now-inaccurate comment above `ENDLESS_TIERS`, redundant guest messaging on the daily
win, and a "Shared!" label that's slightly wrong on desktop. No compile-breaking or
logic-breaking issues found — `tsc --build --force` is clean, and the new
`@devvit/web/client` imports resolve.

## High-level view

The correctness of the two-generator split rests entirely on object-spread order, and
both functions get it right. The pool generator (`generateEndlessPuzzle`) spreads
`...config` *after* `requireUnique:false, rejectInert:true, maxStates:40_000`, so an
easy tier's `requireAllPiecesUsed:true` survives (trap-free), while medium/hard keep
the no-clutter gate and drop uniqueness. The fast fallback (`generateEndlessFallback`)
spreads `...config` *first* and then forces every gate off, so even easy's
`requireAllPiecesUsed` is overridden — all gates off, as intended.

`maxStates` is a new `GenOption` defaulting to `60_000`, the exact constant previously
hardcoded at both solver call sites. Callers that pass nothing (daily, tests) are
byte-identical to before; only the endless functions lower it. Backward-compatible.

The `refillEndlessPools` change adds a per-tier cap (`ceil(maxGenerate/3)`) so one run
tops up every tier instead of draining the whole budget into easy, and the install
warm-up was bumped 6→12 to match. This genuinely closes the "pick Medium right after
install and hit an empty pool" gap.

The client failure path now fails fast (a 12s `AbortController` timeout) into a retry/
menu recovery UI rather than hanging on the loading screen, and `updateHud` clears that
UI on the success path. The sign-in CTA is gated on `context.username` and only mounted
on win screens (because `showLoginPrompt` reloads the page). The share button prefers
the native share sheet and falls back to the clipboard so the paste-into-comments flow
still works.

The symbol rename is clean: `generateEndlessPuzzle` is still used by
`refillEndlessPools`, `generateEndlessFallback` is exported and imported by `api.ts`,
and no dangling import remains. `src/shared` stays pure, no `as` casts were added, and
the diff uses `type` aliases throughout.

### Two endless generators, separated by spread order

The freeze fix splits endless generation in two. `generateEndlessPuzzle` is the pool
generator (scheduler cron + install warm-up), `generateEndlessFallback` is the
request-path generator used only when the pool is momentarily empty. The behavioral
difference is entirely in which `generate()` gates are active, and that is controlled
by where `...config` lands in the options object.

Pool generator (endless.ts):

```
requireUnique: false,
rejectInert: true,
maxStates: 40_000,
...config,            // easy: { ..., requireAllPiecesUsed: true }
attempts: 500,
```

`...config` is spread after the gates, so an easy tier's `requireAllPiecesUsed:true`
takes effect (trap-free), `rejectInert:true` stays on for all tiers (no-clutter), and
`requireUnique:false` stays off (uniqueness dropped — endless is solo). Medium/hard
configs carry no gate flags, so they inherit `rejectInert:true` + decoys allowed.

Fast fallback (endless.ts):

```
...config,            // easy: { ..., requireAllPiecesUsed: true }
requireUnique: false,
rejectInert: false,
requireAllPiecesUsed: false,
maxStates: 15_000,
attempts: 150,
```

Here `...config` is spread first and then every gate is forced off, so even easy's
`requireAllPiecesUsed` is overridden to `false`. All gates off, smaller attempt budget,
tighter solver cap — it trades polish for never stalling the request. The asymmetry in
spread order between the two functions is deliberate and both directions are right.

The one stale artifact: the comment above `ENDLESS_TIERS` still claims medium/hard
"keep uniqueness," which was true of the old `generateEndlessPuzzle` (it had
`requireUnique:true`) but no longer is.

### `maxStates` threading is backward-compatible

`maxStates` is a new optional `GenOption` with `?? 60_000`, and the two solver calls in
`generate()` (`solve(board, { maxStates })` and `countShortestSolutions(board, {
maxStates, cap: 2 })`) now read it instead of a literal `60_000`. Since the default
equals the old constant, any caller that doesn't pass `maxStates` — the daily, the
tests — behaves exactly as before. Only the endless generators lower it (40k pool, 15k
fallback).

### Budget spread fixes the warm-up gap

`refillEndlessPools` previously walked the tiers and let the first one (easy) consume
the whole budget, so a 6-puzzle install warm-up could leave medium/hard empty. The fix
adds `perTier = ceil(maxGenerate / 3)` and caps each tier's `need` by it, and the
install warm-up was bumped 6→12 so each tier gets ~4. A single run still can't fill an
empty tier all the way to `POOL_TARGET=12` (it caps at 4/run), but the fast fallback
covers the gap until the cron catches up, which is the documented design.

### Load-failure recovery and request timeouts

`fetchWithTimeout` wraps `fetch` with a 12s `AbortController` so a stalled daily/endless
request rejects instead of leaving the player on "Getting the penguins ready…" forever;
the rejection lands in the existing catch, which now calls `showLoadError` instead of
just rewriting the loading text. `showLoadError` mounts a retry button (re-runs
`loadPuzzle`) and a menu button, tracked in `loadErrorViews` and torn down by
`clearLoadError`, which runs at the top of `loadPuzzle`, inside `showLoadError` itself,
and in `updateHud` on the success path.

Two small inconsistencies, neither breaking: `loadErrorViews` is the only transient
field not reset in `init()` (it leans on `clearLoadError()` running early in
`loadPuzzle`), and the `resize` handler repositions `loadingText` but returns before it
could reposition the retry/menu buttons when `board` is still null, so on a resize
mid-error those buttons keep stale coordinates.

### Sign-in CTA placement and win-screen density

`addSignInPrompt(y, label)` early-returns when `context.username` is set (consistent
with the existing `context.username` reads in `onWinCommunity` and `splash.ts`), and is
only invoked from the two win screens because `showLoginPrompt` reloads the page. The
daily placement (`h*0.66`) sits comfortably between the status line (`h*0.56`) and the
share button (`h*0.77`).

The endless placement is the concern. `onWinEndless` already stacks a lot vertically,
and the sign-in button is added at `h*0.61`:

```
counts branch:  moves h*0.44 · "Solved:N" h*0.54 · [sign-in h*0.61] · Next h*0.68 · Level h*0.8
hint branch:    note h*0.36 · sub h*0.46 · "Solved:N" h*0.56 · [sign-in h*0.61] · Next h*0.68 · Level h*0.8
```

In the hint branch the count text (`h*0.56`) and the sign-in button (`h*0.61`) are only
`0.05h` apart; on a ~540px-tall webview that's the count text's bottom essentially
touching the button's top, and anything shorter overlaps. The counts branch is a little
roomier at `0.07h`. This only fires for a logged-out player (the CTA is gated), but
art-direction calls for 44-48px touch targets and the hackathon weighs mobile fit
heavily, so it's worth tightening.

There's also redundancy on the daily win: a logged-out player sees both "Playing as a
guest — scores aren't saved." (the status line from `submitSolve`) and the "Sign in to
save your streak" button. Complementary, but it's the same message twice.

### Share sheet with clipboard fallback

`copyResult` now tries `showShareSheet({ title: 'Ice Hop', text })` first and only falls
back to `navigator.clipboard` if it throws, so the paste-into-comments flow survives on
platforms without a share sheet. `showShareSheet`'s signature accepts optional
`title`/`text` and returns `Promise<void>`, so the call typechecks. One copy nit: the
share sheet "copies to clipboard depending on device" (per its own doc), so on desktop
the button reads "Shared!" when it actually copied — slightly misleading, though
harmless.

### Symbol references and Devvit constraints

`generateEndlessPuzzle` is no longer imported by `api.ts` but is still used by
`refillEndlessPools` in the same module, so the export is live and nothing dangles;
`api.ts` imports `generateEndlessFallback` and uses it on the pool-empty branch. The new
client imports (`showLoginPrompt`, `showShareSheet`) both resolve through
`@devvit/web/client` → `@devvit/client/effects/*` and are both used. `generator.ts`
imports only from `../game/types`, `./solver`, `./quality`, so `src/shared` stays pure.
No `as` casts were added in the diff, and no `interface` declarations. `tsc --build
--force --pretty false` produces zero diagnostics, so there are no unused-locals or
type errors in the changed files.
