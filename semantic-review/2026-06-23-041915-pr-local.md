# Scrollable, masked "My puzzles" list with board thumbnails

This change turns the static, row-capped "My puzzles" view into a drag/wheel-scrollable list and gives every row a tiny code-drawn board thumbnail (`drawBoardThumb`) so a creator can recognise each puzzle at a glance. Rows now live in a single container that is moved within a viewport and clipped by a geometry mask; scroll state and input handlers are registered once in `create()` and read the live bounds on each event. The approach is sound, and every runtime gotcha the author explicitly worried about — member/Scene collisions, `make.graphics` on the display list, scroll clamping, listener accumulation, drag-vs-tap, destroy lifecycle, and the thumbnail index mapping — is handled correctly. The one serious problem is the clipping mechanism itself.

**Watch for:**
- **Viewport clipping is a no-op under WebGL (confirmed).** The game boots with `type: AUTO` (game.ts), which selects the WebGL renderer on essentially every device. In Phaser 4, `GameObject.setMask` warns and returns without applying under WebGL — geometry masks are Canvas-only there. So `layer.setMask(...)` does nothing: rows are not clipped, they render over the title/subtitle/Menu pill/"Build another" footer and spill off-screen, and `create()`/`build()` log a console warning each time the list is built.
- **Scroll resets to top on every rebuild/resize (confirmed, low).** `buildList` sets `layer.y = scrollTopY` unconditionally, so any resize (mobile keyboard, rotation, Devvit viewport change) snaps the list back to the top.
- **Thumbnail backing is always square (confirmed, cosmetic).** Non-square boards get uneven ice padding because the backing rect is `size×size` while the cell grid is letterboxed and centred.

## High-level view

The scroll feature is built on `Container.setMask` + a `make.graphics()` stencil, which is the correct Canvas idiom but is silently inert under the WebGL renderer the game actually runs. Everything downstream of "the mask clips the list" therefore fails on the default renderer: the list is unbounded. This is the only behavioral defect; it is also the highest-impact one because it defeats the stated purpose of the rewrite ("clipped to its viewport so rows never overlap the header or footer").

The member-naming audit comes back clean. None of the new fields or methods shadow a property Phaser injects into a Scene, and the earlier `load` → `loadPuzzles` rename was the right fix — `load` is the Loader plugin's injection name. `make.graphics()` is correctly used (not `add.graphics()`), so the stencil never joins the display list and there is no stray white box.

The scroll arithmetic is correct and cannot overscroll: the clamp pins `layer.y` to `[scrollMinY, scrollTopY]`, and both drag and wheel move in the natural direction. Input is registered once in `create()` and reads live bounds, so rebuilds don't stack listeners; on scene exit the per-scene InputPlugin tears its own listeners down and the global `scale` "resize" listener is removed by hand, so re-entering the scene doesn't accumulate handlers. Drag only engages inside the viewport band, which sits between the two pills, so the Menu and Build buttons keep working as taps.

The destroy/rebuild path is also sound: rows are children of `layer`, and destroying `layer` destroys them (Container's `preDestroy` runs `removeAll(exclusive=true)`); the mask graphics is destroyed exactly once via `content`; nothing is double-destroyed or used after destroy. The thumbnail's `row*cols+col` indexing matches the engine's `row*width+col`, so it is correct for non-square boards too; only the square backing is a cosmetic nit.

## Viewport clipping is a no-op under the WebGL renderer

This is the load-bearing problem. `buildList` clips the scrolling column like this:

```ts
const maskG = this.make.graphics();
maskG.fillStyle(0xffffff);
maskG.fillRect(0, top, w, vpH);
layer.setMask(maskG.createGeometryMask());
```

Two separate facts decide whether this works, both verified against the installed `phaser@4.1.0` source.

First, the good half (this answers the author's white-box question): `make.graphics()` does **not** add the stencil to the display list — `GraphicsCreator` only calls `displayList.add` when `config.add` is truthy, and here no config is passed. So the mask graphics never renders as a white box; using `make` rather than `add` is exactly right.

The bad half: `setMask` is implemented by the `Mask` component, whose source begins with

```js
setMask: function (mask) {
    if (this.scene.renderer.type === CONST.WEBGL) {
        console.warn('...setMask: This method is not supported in WebGL. Create a Mask filter instead.');
        return this;
    }
    this.mask = mask;
    return this;
}
```

Under WebGL it logs a warning and returns **without setting `this.mask`**. Geometry masks via `setMask` are Canvas-only in Phaser 4; the WebGL path is the filter system (`FilterList#addMask`), as both the warning and the Container class docs ("masks are filters … In Canvas rendering, only the Container's mask will be applied") state.

Which renderer runs? game.ts:

```ts
const config = { type: AUTO, /* … */ };
```

`Phaser.AUTO` picks WebGL whenever a WebGL context is available and falls back to Canvas only when it is not. The Reddit webview is modern Chromium with WebGL, so the game runs WebGL and `renderer.type === WEBGL`. The mask is never applied: `layer` is unclipped.

Consequences on the default renderer:
- The list is not bounded to `[viewportTop, viewportBottom]`. Rows render over the title, the totals subtitle, the "‹ Menu" pill, and the "Build another" footer, and any row scrolled past an edge stays drawn instead of being hidden.
- `create()` (and every `build()` on resize) prints the `setMask` warning to the console.
- The scroll math still runs and `layer.y` still moves, so dragging "works," but with nothing clipped it just slides a tall stack of cards over the whole scene.

This won't surface in type-checking (the API exists and is typed) or on a Canvas fallback device — it needs a WebGL run. Fix by clipping in a WebGL-compatible way: the Phaser 4 filter API (`layer.enableFilters()` then the filter list's `addMask`), a dedicated Camera whose viewport rectangle is the `[top, bottom]` band, or the previous approach of only instantiating the rows that fit.

## Member names vs Phaser's injected Scene properties

The naming audit comes back clean. Phaser injects a fixed set of names onto a Scene instance (from `InjectionMap`): `game, renderer, anims, cache, plugins, registry, scale, sound, textures, events, cameras, add, make, scene, children, lights, data, input, load, time, tweens, physics, matter` — plus `sys` and the lifecycle methods `init/preload/create/update`. `children` is real (it maps to the display list) and `load` is the Loader, which is why the prior `load()` method crashed `create()`; `loadPuzzles` is the correct resolution.

None of the new fields (`listLayer, viewportTop, viewportBottom, scrollTopY, scrollMinY, dragging, dragPointerStart, dragLayerStart`, plus existing `bgLayer, content, subs, totals, loaded, loadError`) or methods (`scrollable, clampScroll, onPointerDown, onPointerMove, onPointerUp, onWheel, loadPuzzles, build, buildList, buildRow, buildErrorState, buildEmptyState`) collide with that set. `content`, `build`, and `loaded` look closest to a framework name but none are injected. Overriding `create` is intended.

## Scroll math, overscroll, and reset-on-rebuild

The clamp is overscroll-proof. With `clampScroll(y) = max(scrollMinY, min(scrollTopY, y))`, `scrollTopY = top`, and `scrollMinY = contentH > vpH ? top - (contentH - vpH) : top`, `layer.y` is always pinned to `[scrollMinY, scrollTopY]`. `contentH = subs.length*step - SPACE.sm` is the exact bottom of the last row (rows centred at `i*step + ROW_H/2` ⇒ last bottom `= n*step - SPACE.sm`), so scrolling to `scrollMinY` lands the last row flush with the viewport bottom rather than cutting it off. Drag (drag up → `pointer.y` down → `layer.y` down → later rows revealed) and wheel (`layer.y - dy`) both move naturally.

The low-severity behavior is the reset: `buildList` always sets `layer.y = scrollTopY`, and `build()` re-runs on `scale 'resize'`, so any viewport change (mobile keyboard, orientation, Devvit re-layout) jumps the user back to the top. Preserving the prior `layer.y` re-clamped to the new bounds would keep their place.

## Input registration across rebuilds and scene re-entry

Handled correctly, and the asymmetry is the interesting part. The five pointer/wheel handlers are registered once in `create()`, never in `build()`, so repeated `build()` calls (on load and resize) don't stack listeners — handlers read `this.listLayer` and the bounds live, so they don't need re-binding.

Across scene exit/re-entry (`fadeToScene` → `scene.start`, which shuts the outgoing scene down) there's no accumulation either, for two different reasons the code gets right. `this.input` is a per-scene plugin whose `shutdown` removes its own listeners automatically, so the `this.input.on(...)` registrations need no manual cleanup. `this.scale` is a game-global emitter that is *not* cleared on scene shutdown — and that listener is explicitly removed via `this.events.once('shutdown', () => this.scale.off('resize', onResize))`. Cleaning up the global one by hand while relying on auto-cleanup for the per-scene one is the correct split.

Robustness edge: `dragging` clears on `pointerup` and `pointerupoutside`, covering releases over and outside the canvas. If a press is released entirely outside the browser window such that neither fires, `dragging` stays true and a later move could resume scrolling without a button down. Rare, and not specific to this scene.

## Drag vs tap and the pills

`onPointerDown` only starts a drag when `pointer.y` is within `[viewportTop, viewportBottom]`. The "‹ Menu" pill sits at `~h*0.065` (above `viewportTop`, which is below the subtitle) and "Build another" sits below `viewportBottom` (`= buildBtn.y - buildBtn.height/2 - SPACE.md`), so a press on either pill is outside the band and never starts a drag — the pill's own click runs normally. The rows themselves aren't interactive, so there's no in-band tap target competing with the drag.

## Rebuild/destroy lifecycle and the mask graphics

Safe. `build()` does `this.content.forEach(o => o.destroy())` then rebuilds. Rows are added to `layer` (via `layer.add(...)`) and only `layer` is pushed to `content`; destroying `layer` destroys the rows because `Container.preDestroy` runs `removeAll(!!this.exclusive)` with `exclusive` defaulting to true. The nested thumbnail container and its graphics go the same way. No orphaned row objects.

The mask graphics is pushed to `content`, so it's destroyed exactly once per rebuild — no double-destroy (it isn't also a child of `layer`) and no use-after-destroy: `content` is ordered `[…, layer, maskG, …]` so `layer` is torn down before `maskG`, and rendering doesn't happen during the synchronous destroy-then-rebuild inside `build()` anyway. The `GeometryMask` wrapper isn't explicitly destroyed, but it only holds a reference that is GC'd. (Under WebGL the mask was never applied, so `maskG` is created and destroyed each build for nothing — harmless, but more dead weight from the masking approach.)

## Board thumbnail: index mapping and non-square boards

`drawBoardThumb` computes `idx = r * cols + c` with `cols = board.width`, matching the engine's canonical `row * width + col` numbering (types.ts) used by both `board.holes` and `piece.cells`. So holes and pieces land in the right cells for non-square boards (width ≠ height), not just 5×5 dailies, and `cell = size / Math.max(cols, rows)` keeps the grid inside `size` on the long axis.

The cosmetic nit: the ice backing is a fixed `size × size` rounded rect while the cell grid is `cols*cell × rows*cell` centred via `ox/oy`, so a non-square board is letterboxed inside the square backing with uneven empty ice. Fine for square boards; loose for non-square. If UGC boards can be non-square, sizing the backing to the actual grid extent would tighten it.
