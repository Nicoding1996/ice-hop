# Community puzzles as feed posts + a "Comment my score" button

This change turns every accepted UGC puzzle into its own Reddit feed post and adds a one-tap "Comment my score" action to the daily and community win screens. On submit, `submitPuzzle` now also calls `createUgcPost`, which does `reddit.submitCustomPost({ entry: 'default' })`, stores a `post -> submission` mapping under a new `keys.ugcPost` namespace, and seeds a credited how-to comment. `GET /api/init` becomes a discriminated union (`kind: 'daily' | 'ugc'`): a UGC post resolves to its puzzle, and `?daily=1` forces today's daily so the hub can reach it from inside a UGC post. The splash previews a UGC board inline (SVG, no Phaser) with a creator byline and a Build button; `GameScene` plays a `kind:'ugc'` post as a community puzzle; and `POST /api/comment-score` posts the player's result into the current post's thread as the user (`runAs:'USER'`, new `SUBMIT_COMMENT` scope).

Watch for: the comment-score endpoint resolves the puzzle from `context.postId` (the hosting post) rather than the puzzle actually solved, so community-stream solves and forced-daily solves comment with the wrong par, wrong/absent creator credit, and into the wrong thread (confirmed). Submit now blocks on a Reddit post + comment + distinguish round-trip on the request path (confirmed), and a redis failure mid-`createUgcPost` can leave a UGC post that silently serves the daily puzzle (likely). The InitResponse refactor itself is clean and all three consumers narrow correctly (confirmed).

## High-level view

The headline correctness problem is in `/api/comment-score`. It re-derives the puzzle (par + creator) from `context.postId`, which only matches the puzzle on screen when the player opened a post and played *that post's own* puzzle. Every in-app navigation that diverges the played puzzle from the host post — solving a community-stream puzzle while embedded in the daily post, tapping "Next" within a UGC post, or "Back to daily" from inside a UGC post — produces a publicly visible comment with the wrong par, misattributed creator credit, and posted into the wrong thread. The button is shown in exactly those cases.

The submit-to-post coupling is synchronous. `createUgcPost` is awaited inside `submitPuzzle` before the response returns, so the player's "Submit" now waits on `submitCustomPost` + `submitComment` + `distinguish`. The failure handling protects the already-accepted submission (best-effort, never throws out), but the post mapping and the submission's `postId` are written in separate steps, and a failure between them leaves a feed post that `/api/init` can't resolve — falling through to serve today's daily inside a "puzzle by u/X" post.

The `InitResponse` discriminated union is the clean part of the change: a shared `type:'init'` with a `kind` discriminant, daily-only fields (`date`, `board`, `par`, `subscribed`, `solvedResult`) moved onto the daily variant and the UGC payload nested under `puzzle`. All three readers (`GameScene.loadPuzzle`, `splash.ts`, the server constructor) branch on `kind` before touching variant fields, and the type-check is clean.

Two smaller things: the splash now fires `/api/init` on every feed render even for daily posts (where the result is unused), nudging against the "keep splash light" rule; and the win screens gain buttons — the daily bottom-anchored stack grows to 3-4 and squeezes the async leaderboard band, while the community column grows to 4 and can spill past the card on short/landscape viewports.

## Comment-score derives the puzzle from the host post, not what was played

`/api/comment-score` (api.ts) determines par and creator like this:

```
const ugcId = await redis.get(keys.ugcPost(postId));   // postId = context.postId (the HOST post)
if (ugcId) { ...par = sub.par; creator = sub.creator }  // treat as that post's UGC puzzle
else       { ...par = daily(postDate(postId)).par }      // treat as that post's daily
await reddit.submitComment({ id: postId, text, runAs:'USER' });  // always the HOST post's thread
```

`context.postId` is the post the webview is embedded in. It is fixed for the session and does not change as the player navigates scenes. The client never tells the server which puzzle it actually solved (`postScore` sends only `{ moves }`). So the comment is correct only when the host post's mapped puzzle *is* the puzzle on screen. The button is offered in several flows where that does not hold:

```
host post (context.postId)     played puzzle           comment result
---------------------------    --------------------    --------------------------------
daily post                     daily                   correct
UGC post A                     UGC puzzle A            correct
daily post                     community-stream B      par=daily, creator=none,  -> daily thread   WRONG
UGC post A                     community-stream B       par=A,    creator=A,     -> A's thread     WRONG
UGC post A                     today's daily (?daily=1) par=A,    creator=A,     -> A's thread     WRONG
```

Concrete reachable paths (confirmed):

- Open the daily post, go to the hub, "Community puzzles", solve one. `onWinCommunity` shows "Comment my score". The server sees the daily post id, so it comments "Solved today's puzzle in N moves (par <dailyPar>)" into the *daily* thread — for a community puzzle the player solved. Wrong par, wrong thread, no creator credit.
- Open a UGC post, solve it, tap "Next puzzle". `nextCommunity` hands off to a *different* community puzzle B (still embedded in post A). Solve B, tap "Comment my score": the server credits puzzle A's creator and A's par, into A's thread. This is the most common community case — every "Next" desyncs the host post from the played puzzle.
- Open a UGC post, "Back to daily" (now passes `{ daily: true }`). The forced daily loads, but `context.postId` is still the UGC post, so `keys.ugcPost` resolves non-null and the daily solve is commented as "Solved u/<creator>'s puzzle in N moves (par <ugcPar>)" into the UGC thread.

The daily branch in `onWin` adds `postScoreButton` unconditionally, and `onWinCommunity` adds the score button whenever `!isMine` — neither checks whether the played puzzle is the host post's own puzzle. A safe gate would be to only show the button when the puzzle came from the host post (e.g. set a flag in the `kind:'ugc'` load path and on a non-forced daily load, and clear it for stream/forced navigation), or to send the played puzzle's identity and have the server resolve par/creator/target post from that. Severity: high — it emits public, wrong, and misattributed content rather than failing.

## Submit now blocks on Reddit post creation, and a partial failure serves the daily

`submitPuzzle` (ugc.ts) awaits `createUgcPost(submission)` on the request path after the submission is already stored:

```
try {
  const postId = await createUgcPost(submission);                 // submitCustomPost + submitComment + distinguish
  await redis.set(keys.ugcSubmission(id), JSON.stringify({ ...submission, postId }));
} catch (error) { console.error(...) }   // best-effort: never fails the accepted submission
```

A posting failure does not corrupt the accepted submission — it is stored before this block and the catch only logs (confirmed; this is the explicit question about whether posting can corrupt an accepted submission, and it does not). Two observations on the added behavior:

Latency (confirmed coupling): the "Submit" response now waits for a custom-post creation plus a comment plus a distinguish call — several sequential Reddit round-trips folded into the user-facing submit. `createUgcPost` swallows comment/distinguish failures internally, so only `submitCustomPost` and the first `redis.set` can throw out, but the happy path still serializes all of them before the client hears back. The editor's submit UI gives no sign it now depends on post creation.

Partial-state fallthrough (likely): the forward mapping `keys.ugcPost(post.id)` is written inside `createUgcPost`, and the submission's `postId` back-reference is written by the separate `redis.set` in `submitPuzzle`. If `submitCustomPost` succeeds but the `keys.ugcPost` set throws, a real "Ice Hop — a puzzle by u/X" post exists with no mapping. Opening it runs `/api/init`, which finds no `ugcPost` entry, then no `postDate` entry, and falls back to `date = todayUtc()` — so the community-puzzle post silently shows today's daily. The same fallthrough means any unmapped post serves the daily.

On the key-collision question: `keys.ugcPost` (`post:ugc:{id}`) is distinct from `postDate` (`post:{id}`); post ids carry no colons and a post is either daily or UGC, so the two namespaces cannot collide (confirmed).

## InitResponse: the `solved` flag is unused on the UGC arm

The refactor itself is consistent (covered above). One asymmetry: the UGC arm carries `solved`, computed server-side (`typeof playedScore === 'number'`), but `GameScene`'s `kind:'ugc'` branch ignores it — a community puzzle you already solved loads straight into a replay with no recap (the recap is daily-only). The splash does use `solved` to switch its button to "Play again". Consistent with the stated design; noted so it reads as a choice, not an oversight.

## Splash fetches /api/init on every feed render

`splash.ts` runs `void loadPostContext()` unconditionally, fetching `/api/init` for every impression of any Ice Hop post (confirmed). For a daily post the response is fetched and discarded (the `kind === 'ugc'` guard is false) — a wasted round-trip on the inline feed view the tech rules ask to keep light. It is async and does not block paint, so the cost is server-side load. Edge case: for a daily not yet generated, this call runs `getOrCreateDailyPuzzle` and can trigger generation from the splash path.

## Win-screen button density

Daily `onWin`: the bottom-anchored stack is now `[morePuzzlesButton, copyButton, postScoreButton]` plus an optional sign-in/join CTA — up to four pills stacked up from the bottom via `stackFromBottom`, with the async score/leaderboard `status` block filling the band between the moves tally and the top of the stack. `positionWinStatus` shrinks the status text to fit (floor 0.62), so it degrades rather than overlapping in the common case, but each added button lowers `stackTop` and narrows the band; on very short viewports the band can collapse to near-zero — the shrink guard is `if (band > 0 ...)`, so a non-positive band leaves the status unscaled and it can crowd the moves line / buttons.

Community `onWinCommunity`: for another player's puzzle the column is now `[voteButton, scoreButton, moreButton, dailyButton]` laid out downward from `h*0.52` via `stackColumn` (which grows down from `top`, it does not center). Four pills from 0.52h plus the optional stats note at 0.44h can push "Back to daily" below the card panel on short/landscape viewports — the score button is the one that tips it over. Portrait mobile, the target, has room.

## Double-post protection is client-only; move count is unverified

`postScore` guards with `this.scorePosted`, reset per puzzle in `init()`. That blocks a double-tap within one win screen (confirmed), but the in-code claim that the action "can't double-post" overstates it: the server has no idempotency, so re-entering the scene (replay a daily, or any fresh scene start) resets the flag and a second identical comment can be posted. Separately, `/api/comment-score` trusts the client's `moves` outright (only `Number.isFinite && > 0`); a crafted request can comment any move count without solving. Both are low severity — the comment is authored by the user as themselves — but the spoiler-free, server-composed text is the only integrity control here.

## Build intent and dead data

The build-intent handoff (`splash.ts` sets `localStorage 'icehop.launch'='build'`, `Boot.ts` reads-and-clears it) is read-once and cleared on read, so a stale flag can't misroute a later launch; the only residual edge is a flag set by Build that never opens the expanded view, which a subsequent auto-open would consume once into the editor — minor. Separately, `UgcSubmission.postId` is written (`{ ...submission, postId }`) but never read (`\.postId` has no matches) — dead data today, and the natural hook a correct comment-score target would use.

## Test coverage

No tests were added. `buildScoreComment` is pure and a natural unit (par/over-par/multi-over wording, star clamp, creator vs no-creator phrasing) but is uncovered — `share.test.ts` only exercises `buildShareText`. The new `/api/comment-score` resolution and the `submitPuzzle -> createUgcPost` path are untested, including the cross-post mismatch above, which a small test asserting "comment text/target is chosen from the played puzzle, not `context.postId`" would have caught. `community.ts` ordering and the solver/validate suites are unchanged and still apply.
